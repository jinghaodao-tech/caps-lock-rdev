# caps-lock-rdev

WutoHotkeyの代替として、Rust (`rdev`) を用いたキーボードランチャー。IME干渉を回避するため、F13キー（未使用キー）をトリガーにして特定のアプリケーションやサイトを起動する。

## 特徴
*   **低レイテンシ**: Rustで実装されているため、OSレベルのキーイベントを高速に処理します。
*   **カスタマイズ性**: コードベースで直感的にキーバインドや起動するアプリケーションを編集可能です。
*   **軽量**: 余計なGUIを持たず、バックグラウンドで最小限のリソースで動作します。


##  技術スタック
*   **Language**: Rust
*   **Library**: [rdev](https://github.com/Narsil/rdev) (キー入力のキャプチャ)
*   **Platform**: Windows

## 解決した技術課題


### 1. IME干渉の回避

- **手法**: Windowsのキーボード拡張(PowerToysを使用)として一般的なF13キーをトリガーとして使用。
    
- **課題**: 常時常駐させるために `rdev` を使用し、キー入力イベントを低レイヤーでキャプチャ。

###2. キーの押しっぱなし判定（F13_PRESSED）
　
　導入理由：無事に動いたが指を離しても判定が残ったままだったため特定のキーを押すと意図せず挙動してしまったため。

- **手法**: `static mut F13_PRESSED: bool` を定義し、キーの状態をグローバルに管理。
    
- **工夫**: 特定のキー（W, T, Q, V）が押された瞬間に `F13_PRESSED = false` を実行することで、ショートカット発動後の「戻し忘れ」による誤入力を強制リセット。

### 3. スペース入りパスの解決（最難関）
　
　obsidianは.lnk経由で簡単に結びつけることができたがVSCodeはターミナルが開いてしまい、それを直そうとするとVSファイルが見つからないと表示され、今度はCodeファイルが見つからないとされ、スペースがパスに含まれるものは導入が困難であった。

- **事象**: `Visual Studio Code` のようにパスにスペースが含まれると、Windowsがパスを正しく解釈できず「ファイルが見つかりません」エラーが発生した。
    
- **解決策**:
    
    - `cmd /C start` コマンドを使用。
        
    - さらに、長いパス名（`Microsoft VS Code`）を **「短いパス名（8.3形式）」** に変換して記述することで、パスの途切れ問題を完全に回避。



**課題：PowerShell呼び出しによるチラつき**
    
    - 当初は `Command::new("powershell")` を使っていたが、実行のたびに一瞬コンソールが表示される。これはユーザー体験を著しく悪くする
        
    - **解決**: `windows-rs` を導入し、Rust のコードから直接 Windows API を叩く実装へ刷新。これによりサブリミナル的な画面表示をゼロに成功。
        
**課題：ビルド時のファイルロック問題**
    
    - 実行中のプログラムを再ビルドしようとすると `Access Denied` になる問題。
        
    - **解決**: `taskkill` を活用したビルドフローの確立と、管理者権限での実行、および適切なプロセス終了処理の徹底。
        
**課題：Rustの所有権とWindows APIの型変換**
    
    - `EnumWindows` に文字列を渡す際、所有権の移動によるエラーが発生。
        
    - **解決**: `LPARAM` 型への安全なポインタ変換と `clone` の適切な使用により、メモリ安全性とAPI要件を両立
